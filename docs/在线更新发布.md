# 在线更新发布

PeakHalo 使用 Sparkle 做应用内更新。用户在设置的 About 页面点击检查更新后，Sparkle 会读取 `SUFeedURL` 指向的 appcast，下载 ZIP 更新包，校验签名，然后安装并重启应用。

## GitHub Secrets

发布前需要在 GitHub 仓库配置两个 secret：

- `SPARKLE_PUBLIC_ED_KEY`：写入 app 的 `SUPublicEDKey`。
- `SPARKLE_PRIVATE_ED_KEY`：仅在 GitHub Actions 中用于签名 `appcast.xml`，不要提交到仓库。
- `CODESIGN_IDENTITY` 或 `DEVELOPER_ID_APPLICATION`：Developer ID Application 签名身份，例如 `Developer ID Application: Example Team (TEAMID)`。

生成密钥：

```bash
.build/artifacts/sparkle/Sparkle/bin/generate_keys
```

`generate_keys` 会输出要写入 `SUPublicEDKey` 的公钥，并把私钥保存到本机钥匙串。CI 需要私钥字符串时，可以按 Sparkle 文档导出后保存到 `SPARKLE_PRIVATE_ED_KEY`。

## Release Flow

打 tag 后 `.github/workflows/build-macos-package.yml` 会执行：

1. 构建 `PeakHalo.app`。
2. 生成 ZIP、DMG、PKG。
3. 使用 ZIP 生成 Sparkle `appcast.xml`。
4. 上传 ZIP、DMG、PKG、`appcast.xml` 到 GitHub Release。

应用内更新使用：

```text
https://github.com/logyxiao/PeakHalo/releases/latest/download/appcast.xml
```

更新包下载地址由 appcast 指向对应 tag 下的 ZIP，例如：

```text
https://github.com/logyxiao/PeakHalo/releases/download/v0.1.6/PeakHalo-v0.1.6.zip
```

DMG 和 PKG 仍然保留给手动安装；Sparkle 在线更新只使用 ZIP。

## Signing

本地没有签名证书时，`script/package_app.sh` 会使用 ad-hoc 签名。这种包只能做结构验证，不适合作为真实分发包；macOS 可能在启动或在线更新安装时拒绝未知签名链。

正式发布需要使用 Developer ID 证书签名，必要时再做 notarization。打包脚本会读取：

```bash
CODESIGN_IDENTITY="Developer ID Application: Example Team (TEAMID)"
```

或：

```bash
DEVELOPER_ID_APPLICATION="Developer ID Application: Example Team (TEAMID)"
```
